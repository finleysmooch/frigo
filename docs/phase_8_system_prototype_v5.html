Cook post depletion — three options

All three respect your "passive, not forced" direction. Differences are where depletion happens in the flow and how much agency the user has at each step.

Option A · Banner after post (recommended)

  • Cook post flow is unchanged. Post as you would today.
  • After success, a banner appears: "✓ Posted · We updated your pantry · 4 items · review / undo"
  • Silent action by default. Two escape hatches (review / undo).
  • Assumes the common case ("yes, deplete what the recipe called for") and trusts the user to undo if wrong.

Option B · Inline toggle at post

  • One small row at the bottom of compose: "Update pantry · 4 items" with toggle.
  • User sees what's being affected without having to tap through a sheet.
  • Toggle defaults on. Tap the row to expand what will be depleted.
  • More visible than A, but adds a UI element to every post.

Option C · Ask later (post-first)

  • Cook post flow has no depletion UI at all. Post and move on.
  • Next time user opens Pantry, a banner: "You cooked X yesterday · Review / Skip"
  • Most passive. But creates a time gap where pantry state is out of date.
  • If user never opens pantry after cooking, depletion never happens.
Try it: Toggle between A / B / C with the sub-toggle at the top of the phone mockup. Notice how A is the fastest flow (cook → post → done), B is most explicit, C is most deferrable.

My take

A wins for most users — fast, trust-based, reversible. C is great for power users or passive cookers (Mary logs a cook she forgot about, doesn't want to think about pantry). B is the safe middle. If you pick one for v1 I'd do A; if you want to offer choice, Pantry settings could let users switch the default.

Browse recipes — v4 structure

Three zones stacked: search (with mic for voice) at top, tiles in the middle, full scrollable list below with view toggle + filter chips. No more empty feeling.

Try it:
• Tap the search bar → jumps to Natural search tab.
• Tap a tile → filtered drill-in (toast).
• Tap "Scroll all" tile → scrolls to the full list.
• View toggle at the full list: Pantry % / Recent / Rating / Time — user picks what matters.

Why all three coexist

  • Search-led user: types or talks what they want, gets parsed results.
  • Mood-led user: doesn't know exactly what, taps "Quick tonight" or "Something new."
  • Scroll-led user: scrolls past tiles, lands in a sortable list with familiar filter pills.

Tiles stay because they're the fastest answer to "I have a general feeling but no specific query." Raw scroll stays because sometimes you just want to browse. Search elevated because it's the fastest path for users who know what they want — and with natural language, that's far more people than today.

Natural language search — how it works

Type or dictate free text → Claude Haiku parses into structured filter chips → query runs against existing filter engine → results shown with transparent filter representation.

Try it:
• The query "something thai or indian with chicken from ottolenghi…" is pre-filled. Tap Search to see the parsed chips.
• Tap × on any chip to drop that filter and re-query.
• Tap an example prompt at the bottom to see how different queries parse.

Architecture

  • Parse layer (Haiku): free text → structured JSON with known-vocabulary matches (cuisines, chefs, ingredients) + pantry % threshold + free text residual. ~500ms latency. ~$0.0001/call.
  • Execute layer: reuses existing searchService.ts filter object. Same engine as the FilterDrawer.
  • Transparent: user sees exactly what was parsed. Can correct misreads by tapping ×.
  • Graceful fallback: if parse fails or returns empty, falls through to existing text search.

Why this fits Phase 8

  • 1-2 sessions to build (new parse function + prompt + chip UI + existing engine reuse).
  • Low operational cost — Haiku call per search is negligible.
  • Voice support free — iOS/Android system dictation reads into the textarea.
  • Ships a genuine AI moment without scope creep.

What's deferred to post-F&F

  • Conversational refinement — "actually make it under 30 min" as a follow-up query. Requires context persistence.
  • Query suggestions / autocomplete — "did you mean...?" — requires learning from query corpus.
  • Dietary preference injection — searches for Mary auto-add GF filter. Needs user-profile awareness, small add.
  • Voice-native mode — app-level voice recording beyond OS dictation.

V5 changes summary

1. Unknown state on staples

New dashed-border treatment for staples the user hasn't confirmed yet. Tap the dot → confirms as "good" and exits unknown state permanently. See Paprika on the Pantry tab — that's the state. Real-world case: you mentioned always re-buying paprika because you couldn't remember if you had any. Now you'd add it to staples list, leave it in unknown state until you check the shelf, tap to confirm.

2. Out prominence — subtle bump

Out staples now auto-sort to top-left position (Cholula is first). Label text is 500 weight. Tint kept subtle per earlier feedback. Distinction is "slightly more noticeable" not "alarming."

3. Recipe tab — existing layout preserved

Reverted to the Phase 6G NYT-style format: group headers, ✓ qty unit name, preparation flat row format. No bordered cards. Tap-sheet appears inline below the tapped row (not an overlay). Actions adapt to ingredient state — missing gets "Substitute," low staple gets "Actually have" / "Mark out," etc. Implements the D6-18 deferred feature.

4. Locked filter chips pattern

On subset pages (What can I cook, Ingredient Detail Recipes), the page's defining filter shows as a locked chip with a small lock icon — not removable. User-added filters use blue treatment and are removable. Natural-language search bar above the chips accepts refinements that layer on top of locked filters. See What can I cook — "Pantry match ≥ 60%" and "Excludes staples" are locked; any free text search starts with those applied.

5. Browse collapsed filter row

View toggle + filter pills collapsed into a single line ("Sort: Pantry % · 0 filters · Filter & sort ▸"). Tap to expand. Recipes below get more visual weight on initial scroll.

Path B data model prep

Per the decision to move toward Path B eventually:

  • Add last_confirmed_at TIMESTAMPTZ to both pantry_items and pantry_staples tables
  • Add staleness_threshold_days per category to user preferences (defaults: fresh produce 7d, dairy 14d, pantry staple 60d, freezer 180d)
  • Update pantryService functions to bump last_confirmed_at on any state change (qty edit, expiration edit, state cycle, cook-post depletion)
  • F&F ships with unknown state visible only for staples (current prototype behavior) — the underlying data collects during F&F, post-launch we turn on stale treatment for tracked items with 1 session of UI work

Decisions still open

  1. Out position in staples grid. Current: auto-sort to top-left. Alternative: keep position stable, just make bolder. Flipping is a user-disorientation concern if an item you expect at position X moves. My vote: auto-sort is right — if something's out, it should lead. Your take?
  2. Inline tap sheet vs in-place actions. The sheet appears below the tapped row as a panel. Alternative: inline actions replace the row itself, no separate panel (tap cilantro → row swaps to action buttons). Current approach keeps context; the alternative would use less space. Works as-is?
  3. Unknown state entry point. Prototype shows Paprika as existing-unknown. How does a user add a staple in unknown state in the first place? "Add new staple" → three default state options including unknown, or always starts unknown and user confirms on first check?
  4. Locked chip wording. "Pantry match ≥ 60%" / "Excludes staples" — these are specific. Better as plain English? "Only recipes I can make" / "Ignoring staples"?
  5. "Which step?" action in recipe sheet. Implements D6-18. Is that the right name? Alternatives: "Find in recipe," "Jump to step," "Used in steps." What reads most natural?

Scope impact vs v4

Net neutral. v5 simplifies in places (recipe tab doesn't need new overlay sheet — uses existing layout with tap-sheet addition). Path B schema prep (0.5 session) is new. Locked chip pattern is reusable (0.5 session). Net: still 16-23 sessions.
