Pantry overview — updated visual

Changed from v1:
  • Lot badge color reflects status (not just numeric). "0" eggs in red because status=out, "2" coffee in amber because Low, "12" eggs in green because in_stock. The badge is now a status-aware affordance.
  • Tap badge → cycles status just like the 5-circle dots on non-lot supplies. Number doesn't change; color + label do.
  • Right-side text restored to status word ("In stock", "Low", "Out") instead of "4 packs" / "2 lots." Aggregate qty info moved to the meta line.
  • Unit icons inside badge: count dot, bag, pack, container — replacing the emoji set.
Why this works: Status (in_stock/low/critical/out) and lot count are independent dimensions. The badge fuses them visually but the user only manages status via tap. Lots can be edited from the expand panel or supply detail, but day-to-day status flagging works exactly like before.
Q-V1 — eggs at 0: when the lot count hits 0, status auto-flips to out (no manual cycle needed — it's a fact). Badge shows "0 ◯" in red. Confirm: this auto-flip is the only case where lots affect status.

Coffee — lots + status, fully decoupled

Changed from v1:
  • v1's three model variants collapsed to one. Per your direction (Q-W2/Q-W4/Q-W10): "users aren't going to keep active track of how much is left in a bag, but want to click a button saying low or critical."
  • No per-lot fill_level / no usage_level on the lots themselves. Lots are inventory facts. Status is the judgment signal.
  • Tap badge to cycle (in_stock → low → critical → out → in_stock). User signals "we're low" without touching qty.
  • The decision Tom made on Q-W11 holds: supplies.usage_level stays as a column for non-lot supplies (the 5-circle status indicator for olive oil, etc.). For lot supplies, usage_level is unused — status alone drives everything.
What the user does: When they're getting low on coffee, they tap the badge once to flip it to amber/Low. Doesn't matter if there's a sealed bag in the pantry or not — the status is "we should buy more before we're out." When they buy more, they add a lot (or check off a need that auto-creates one). Status flips back to in_stock automatically (auto-restock on lot add — see Tab 10).
Q-V6 — auto-restock on lot add: when a tracks_lots supply is in low/critical/out status and a lot is added, status auto-flips to in_stock. Lean: yes, this matches user expectation ("I just bought more — we're not low anymore"). Edge case: user manually adds a lot but supply is genuinely still low (only added 1 small lot). Solution: show a confirmation toast "Status updated to In stock — undo?" with 5s auto-dismiss.

Search — multi-token AND across all dimensions

Tab 8d is new (per Q-W20). Demonstrates the frozen pizza scenario: search "frozen gluten free pizza" tokenizes into 3 tokens; each must match at least one dimension on the result.
Match dimensions surfaced via match pills:
  • name = matched ingredient.name or plural_name or supply.custom_name
  • family = matched ingredient.family ("Berries" matched "berries")
  • storage = matched display synonym for storage_location ("frozen" → freezer)
  • tag = matched a tag attached to supply or lot ("gluten free" tag)
  • brand = matched brand on supply or lot
  • variant = matched lot.variant_label
Match pills render small + colored to differentiate. Multiple matches join with " · " separator.
Lot-level match highlighting: when only some lots within a supply match, the matching lot has a yellow background and the non-matching lots are dimmed. This makes "show me Mary's gluten-free pizza" answerable at a glance.
For-user mini badges ("M" / "T") are part of the visible meta — when supply or lot has a non-empty for_user_ids, render the corresponding member's initial. For-user is also a search dimension (search "Mary" → matches lots with for_user_ids containing Mary's ID).
Q-V10 — search infrastructure: implementation question, not design. To support this, we need either (a) a tsvector on supplies materialized from all dimensions on write (heavy but performant) or (b) client-side multi-token AND filter on already-loaded supplies + lots (simpler for F&F, scales to ~hundreds of supplies). Lean: (b) for F&F, server-side denormalization post-launch.
Q-V11 — storage synonym map: "frozen" → freezer; "fridge" / "refrigerated" / "cold" → fridge; "pantry" → pantry; "room temp" / "counter" → counter. Static client-side mapping. Confirm completeness, or add others?

Cook depletion — against lots

New surface in v2. Banner pattern matches existing CookDepletionBanner (already shipped pre-8R). The lot-aware behavior is what's new.
Variant A — default (oldest-first):
  • For each ingredient cooked, the system determines the lot to draw from.
  • Default rule (per Q8): oldest-expiring lot whose qty + unit is compatible. Show which lot was picked: "From: 1.25 lb bone-in pack (oldest, exp Jul 5)."
  • Cross-unit: the unitConverter handles oz ↔ lb, cups ↔ ml, etc. (existing service; reused).
  • Cross-lot decrement: if no single lot has enough, draw from oldest first, then next-oldest. Shown as "From: 0.5 lb of bone-in (last) + 0.75 lb of boneless." (Not visualized here — edge case.)
  • "Change" on a row opens variant B picker.
Variant B — override:
  • Multi-select lot picker, scoped to the recipe ingredient.
  • Selecting multiple = cross-lot draw (e.g., user knows they used the boneless ones).
  • Confirm button shows total qty being drawn: "Use selected (1.25 lb)" — must equal or exceed recipe qty.
Q-V12 — auto-archive policy: when a lot reaches qty=0 from depletion, auto-archive (consumed_at = NOW). Per Q12. Confirm: this happens silently in the depletion flow; user can undo via the banner's existing undo affordance.
Q-V13 — non-lot supply in same cook: "Olive oil · 2 tbsp" is a non-lot supply (status-only). Default behavior: usage_level decrements per existing rules (not changed). Just shown alongside lot supplies in the banner.

Grocery acquire → lot create

New surface in v2. Per Q9 confirmation: default-create with reasonable inferred values + edit affordance.
Variant A — default flow:
  • User checks off "eggs" need (qty 12, no unit specified).
  • System creates lot on the linked supply: qty=12, unit="ct" (from ingredient.typical_unit), storage=fridge (supply default), acquired_at=NOW, expires_at=NOW+21d (computed).
  • If supply was in low/critical/out status, auto-flips to in_stock (per Q-V6).
  • Toast at bottom shows what happened with Edit / Undo affordances. 5s auto-dismiss.
  • "Edit lot" opens the sheet in Variant B for fine adjustment.
Variant B — edit lot sheet:
  • Same fields as the create-lot sheet (per Q-V7).
  • Supply is linked and read-only — can't be changed at this point. (To re-route to a different supply, undo + re-acquire.)
Q-V14 — receipt-scan parallel: when receipt scan ships, the same toast/edit-lot pattern can be used per-line. Bulk-confirm acquires N at once with sensible defaults; per-line edit available. Architectural compatibility check: confirm.
Q-V15 — non-tracks_lots supply acquire: if user acquires a need linked to a non-tracks_lots supply (olive oil), no lot is created — just status flips to in_stock + last_confirmed_at bumped. Existing behavior, unchanged. Toast still fires but says "Restocked: olive oil" without lot info.
