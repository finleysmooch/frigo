Variant C — Merged view

Per D8R-Q12: separate at add-time, merge at view-time. "Cream cheese — 12 oz · 2 recipes" with expand-children.

v3 — Recipe combine rule made explicit (audit C4): two needs merge in views when ALL match:
  • Same identity (ingredient_id or custom_name)
  • Same unit (no cross-unit conversion at v1)
  • Same store tags (as set — different stores = different routing intent, don't merge)
  • Same for_user_ids (Tom's yogurt + Mary's yogurt don't merge even if same ingredient)
Recipe attribution does NOT block merge (the whole point: buy once for two recipes). Urgency tags don't block merge (rendered as recipe-attribution chips). Recipe attribution from both recipes is preserved on the merged view row. Captured as D8R-Q28.

Supply detail (D8R-Q22)

Stores + Brands as first-class sections. For-user demoted to settings-row + multi-select sub-sheet (D8R-Q27).

v3 — state-cycle UI consistency note (audit C8): two state-cycle interaction patterns coexist intentionally:
  • List view dot (Tab 7) = tap-to-cycle (one tap advances to next state in the cycle). Compact, fast — for "I just used some" updates.
  • Detail view 4-step strip (this Tab) = tap-to-set (tap any step to set state directly). Precise — for "I want to mark this exactly low/critical/out."
Different mental models for different contexts. Document at CP5 build that the intentional split is a feature, not a bug. Captured as D8R-Q30.

Stores section

Brands section

Free-form list. Captures preference memory. F&F: just a tag list. Per-store brand preferences post-F&F (P8-2).

Supply create flow (NEW v3, D8R-Q33)

New tab — addresses audit C8. The "+ Add new regular" affordance on Tab 10 used to route to add-to-needs, which forced creation of a need. The case "I want to start tracking eggs but Tom + Mary haven't run out yet" needed a supply-create-only path. This tab is that path.

Triggered from

  • Pantry tab "+" button (Tab 7 header)
  • "+ Add new supply" on Expanded Regulars (Tab 10)

Fields

  • What — autocomplete from ingredients table OR custom name (e.g., "Toilet paper")
  • Current state — In stock (default) / Low / Out. If user picks "Out," a need gets auto-spawned on save (consistent with D8R-Q10) — surfaced via inline footnote.
  • Stores — multi-select chips. Auto-prefills from current view context if invoked from a store-filtered view (e.g., from Costco view → store: Costco prefilled).
  • Brands — optional list. Captures preference memory.
  • For — multi-select sub-sheet (Tab 8 Variant C reused).

What this differs from Tab 11

  • No quantity field — supplies don't have quantity in F&F (only state enum per D8R-Q15).
  • No need creation by default — only the supply is created. Exception: if user picks "Out," need auto-spawns on save.
  • No "Save as regular" toggle — this IS the save-as-regular path.
Common case: Tom + Mary realize they should track Greek yogurt. They open Pantry → tap "+" → type "Greek yogurt" → state: In stock → store: Costco → Save. Supply now appears in pantry grid + as regular in Costco view's Regulars zone. No need created. When it later runs out, auto-spawn fires.
For CP1/CP5: save action writes supplies row with status, identity, and tag memberships. If state=out picked, also writes needs row + supply→need link via the same auto-spawn function used for cook-flow depletion.
