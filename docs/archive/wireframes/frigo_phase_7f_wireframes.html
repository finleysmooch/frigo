L3a · co-cook · partner hasn't posted

Co-cook when the partner hasn't posted yet

Tom posted his kombucha session. He tagged Anthony as a cook partner. Anthony hasn't created his own post yet. The result: a co-cook prehead floats above Tom's card explaining the relationship, but no linked group forms (because there's no second card to link to).

Unified pattern: co-cook context and meal event context both surface as a prehead above the card (for N=1 cases) or a group header above a linked stack (for N≥2 cases). The card itself is always a plain cook card with no context embedded in it. Context lives above the card.

Prehead structure (matches group-header):

  • Small round icon on the left (🍳 for co-cook, 🍽 for meal event)
  • Title line ("Tom cooked with Anthony") — bold, dark
  • Meta line ("Anthony hasn't posted yet" or date/location) — small, gray
  • No border, no chrome — reads as a label, not a widget
  • Not tappable in the co-cook case (no separate entity to navigate to); tappable in the meal event case

What happens when Anthony posts later: the feed retroactively forms a linked group (see L3b). Tom's prehead changes from "Tom cooked with Anthony / Anthony hasn't posted yet" to the shared group header "Tom cooked with Anthony / Apr 10 · Portland, OR", and Anthony's card appears below Tom's in the group.

The "Anthony hasn't posted yet" subtitle is optional — we could omit it and just show the main title. I added it as a hint that there's a relationship the viewer will see more of once Anthony participates. If it feels like noise, drop it.

L3b · co-cook · linked pair

Co-cook when both partners have posted

Tom and Anthony both posted about their kombucha session. Because they're linked as cook partners AND the viewer follows both of them, the two posts render as a linked pair — indented right to leave a left gutter, with a thin gray connector line running down that gutter.

Visual mechanics (Strava-style):

  • Group header at the top reads "Tom cooked with Anthony" with a small round icon. Below: "Apr 10 · Portland, OR". Not tappable — co-cook relationships have no separate entity to navigate to.
  • Cards are indented right by ~20px to make room for a 1px connector line in the left gutter. The line runs from the bottom of the group header down through all linked cards, matching Strava's pattern exactly.
  • Connector line is outside the cards, not cutting through them. Subtle gray, barely visible. It's a grouping hint, not a feature.
  • Each card is fully independent — own title, description, photos, stats, likes, comments. Tom's card has "Kombucha batch #2", Anthony's has "Helped Tom with kombucha". They're different posts, independently authored.
  • Card meta lines no longer repeat the co-cook context — it's already in the group header. Meta just shows time + location like normal.
  • Ordering within a group: oldest-first (reads as a narrative — "Tom posted at 4:12, then Anthony posted at 6:03"). The group itself sorts in the overall feed by its most recent member's post time, so the group surfaces to the top whenever any member posts something new. But inside the group, cards are in chronological order from oldest to newest.

Engagement is per-card: yas chef on Tom doesn't affect Anthony and vice versa. Each author has their own social surface.

If a third cook was tagged too: group shows three cards in the gutter, same pattern. Connector line extends to cover all of them.

If the viewer only follows one of them: no group forms. That viewer sees L3a instead (a single card with a prehead saying "[name] cooked with [partner]").

L4 · meal event · solo

Cook with meal event link (no linked group visible)

Andrew brought his carrot dish to Mary's potluck. Mary and Tyler also cooked dishes. But the viewer only follows Andrew — not Mary or Tyler — so no linked group forms. Andrew's card appears solo, with a meal event prehead floating above the card as the only hint that this was part of a bigger evening.

Unified prehead pattern: the meal event affordance uses the same visual shape as the co-cook prehead (L3a) and the meal event group header (L5). One consistent "context above the card" treatment across all three related cases.

Prehead attributes for L4:

  • Tappable — unlike the co-cook prehead, meal events are first-class entities with a detail screen. The whole prehead row is a tap target; lands on the meal event detail.
  • Title: "Dinner at Mary's" (the event name, set by the host)
  • Meta: "Apr 8 · Mary's house, Portland · 4 cooks" — date, location, and the total cook count at the event
  • Icon: cream-toned circle with 🍽 inside, visually distinct from the co-cook 🍳 variant
  • Visually attached to the card below — no gap between the prehead and the card, so they read as a single unit. The top border of the card is suppressed (.attached-to-prehead) to emphasize the attachment.

What the card deliberately does NOT show:

  • Names of other cooks at the meal (beyond the count)
  • Preview of other dishes
  • Names of eaters / attendees
  • Any in-card meal event chip or banner

All of that is one tap away via the prehead. The cook card stays focused on Andrew's contribution.

Header meta line on the card itself is now cleaner: just "Apr 8 at 7:14pm" — no location (since location is in the prehead) and no date redundancy.

L5 · meal event · linked group

Meal event with 3 cook posts visible

Mary hosted a dinner. She cooked lamb. Andrew brought carrots. Tyler brought olive oil cake. The viewer follows all three, so the feed shows a linked group under a single tappable meal event header.

Unified pattern with L3b: same indent-with-gutter-connector visual. The only difference is the header — here it's a meal event (tappable, teal title) instead of a co-cook label (non-tappable, neutral title).

Key simplifications from the previous draft:

  • Per-card meal event chips removed entirely. The group header is the only meal event affordance. Each card is a clean cook card with no in-card chrome about the event. This dramatically reduces visual noise compared to the earlier "chip on every card inside the group" version.
  • No "Hosting" vs "Part of" distinction per card. Mary's role as host is instead shown on her card's meta line: "Apr 8 at 8:45pm · host". A small, quiet affordance rather than a distinct chip style.
  • Meta lines don't duplicate the location since location is in the group header. Just date + time (+ "host" for the host).

Tappability: the entire group header row is a tap target → meal event detail screen. Each individual card is tappable independently → that cook's detail screen.

Ordering inside the group: oldest first, so the narrative of the evening reads top-to-bottom chronologically. In this example: Tyler (6:30pm) → Andrew (7:14pm) → Mary (8:45pm). Current draft has Mary first, Andrew second, Tyler third. I'll fix this in the next iteration if you confirm oldest-first is the right rule.

Degraded cases:

  • If only one cook is visible to the viewer, L5 degrades to L4 (single card with a solo prehead)
  • If two are visible, L5 shows those two in the group and omits the third. The header still says "4 cooks" (total, not visible) — intentional hint that there's more at the meal event detail

L6 · cook detail screen

Cook detail screen

The new screen type that replaces "dish detail" in your mental model. Tapping a cook card (anywhere on the card, or on a photo, or on the title) lands here. This screen is about what Tom did with this recipe on this specific night — not about the recipe itself. Recipe info lives on RecipeDetail (reached by tapping the recipe line).

Structure, top to bottom:

  • Back nav with title "Tom's cook" and Share action (right side)
  • Hero carousel — same shape as the feed card but bigger. If the user tapped through from a specific photo in the feed, that photo is centered here. Dots indicate position in the carousel.
  • Author block — avatar, name, full timestamp (with time-of-day, more precise than the feed card's "Apr 12"), device location. Tappable → Tom's profile.
  • Title — full, large (22px), not clipped.
  • Description — full, not 3-line-clipped like on the feed card. The user gets the full story here.
  • Recipe line — tappable link to RecipeDetail. Shows cookbook page number for future print-reference support ("p. 121").
  • Stats grid — bigger version of the feed stats row. Cream background to set it apart visually. Three stats by default: Cook time, Your rating, Times cooked.
  • Highlights — expanded pill with a short descriptive paragraph underneath (e.g., "You're on a carbonara streak...")
  • Modifications & notes — a cream-toned block showing what the cook changed. Maps to posts.modifications in the data model. Distinct from description because it's specifically about the recipe delta.
  • Your history with this recipe — a small table showing Tom's previous cooks of this same recipe with their dates and ratings. The current cook is highlighted. This is the per your answer #1 for L6 — you said yes to inline cook history.
  • Photos gallery — 3-column grid showing all photos from this cook. "View all" action if there are many (tapped from feed opens this gallery scroll-positioned to the originally-tapped photo).
  • Comments — inline comment list with "Add comment" action. Full thread visible.
  • Sticky engagement bar — pinned to the bottom of the screen, shows yas chef count + comment count + action buttons. Same pattern as Strava's activity detail.

Deferred for later phase:

  • "Related cooks from friends" section at the bottom — per your answer, deferred
  • Personalized lens on the chef name (Mary viewing Ottolenghi shows her history with Ottolenghi recipes) — interesting idea, not blocking
  • Cookbook page number deep linking (tapping page number could eventually open the cookbook's detail page at that section)

Modifications & notes structure open question: I've shown a single block labeled "What I changed." But posts.modifications and posts.notes in the DB are two separate fields. Should they be two separate blocks (modifications + general notes), or one combined block? I'd lean toward two blocks when both exist, collapsed into one when only one exists. Easy to adjust.

L7 · meal event detail screen

Meal event detail screen

Replaces the old MealDetailScreen under the new cook-post-centric model. Reached by tapping the meal event prehead on a cook card (L4) or tapping the group header on a linked meal event group (L5).

Key conceptual shift: this is no longer "Mary's meal post" — it's a shared view of an event that multiple independent cook posts belong to. The event has a host/creator (Mary) who can edit metadata, but the cook posts are independently authored and the dish rows here are just pointers to those individual cook posts.

Structure, top to bottom:

  • Hero — the host's chosen highlight photo. Defaults to the first photo from the host's own cook post; editable by the host only. Other attendees cannot change the hero but can contribute to Shared media below.
  • Event metadata block — title (editable by host), date + time + location, "Hosted by [name]" line. The title feels larger than a card title because this is a full-page detail, not a feed card.
  • Stats grid — Cooks / Dishes / At table counts. Same visual as L6's stats grid but different content.
  • What everyone brought — the dish rows, borrowing the F1e+ pattern. Each row shows a thumbnail, dish name, cook attribution (avatar + name + optional role like "host" or "family recipe"), a star rating pill (if the viewer has rated it as an eater), and per-dish actions (Save / Ask [cook]). Tapping the row goes to that cook's detail screen (L6).
  • Planned dishes — I'm hybridizing per your answer #5 on L7. Dishes that have been planned but not yet posted show up in the dish list with an italic greyed-out name and a placeholder thumbnail. The cook is still attributed. No actions. Once they post their cook, the row upgrades to a real dish row. This is my call — flagging as DEFERRED_DECISION: confirm we want a planned-vs-contributed distinction, or drop it.
  • At the table — the attendees list, including both cooks and non-cooks. Each row shows role ("Host · cooked lamb", "Cooked carrots", "Guest"). Cam is "Cooking Greek salad" because he's a planned-but-not-posted cook.
  • Shared media — a photo gallery that anyone linked to the event can contribute to. Sits below the cook-contribution flow because it's secondary. The hint line explains that shared media is private to the event (doesn't surface on the feed). An "+ Add photo" action in the header and an add-media tile at the end of the gallery.
  • About the evening — event-level comments. Distinct from per-cook comments (which live on L6). The hint line explains the difference. Two separate surfaces, per your answer.

Permissions model (per your answer #2 on L7):

  • Creator/host (Mary) can edit: event title, date, location, attendee list, highlight photo, delete event
  • Any attendee can: add photos to shared media, add event-level comments, link their own cook posts to the event
  • Rating dishes: any attendee who ate the dish can rate it. Ratings are private to the eater — stored per-viewer, never visible to the cook or to other attendees. The pill on each dish row reflects the viewer's own rating, or "Tap to rate" if they haven't rated yet. This carries over from F1e+ (D43/ζ visibility rule) without redesign.

Top-nav action is context-sensitive:

  • Host view: right action reads "Edit" → opens event metadata editor (title, date, location, highlight photo, attendee management)
  • Attendee view (the view shown in this wireframe): right action reads "Add" → opens a sheet with options to add a photo to shared media, add an event comment, or link a cook post to the event
  • Non-attendee viewer (someone who follows an attendee but wasn't at the dinner): no right action, or just "Share"

What got deprecated from the old MealDetailScreen:

  • "Meal post" as a first-class post object — replaced by the meal event record
  • "Add Dish" as an in-screen action — replaced by "each cook adds their own cook post, linked to the meal event"
  • Host-as-sole-author — replaced by creator-as-owner with shared contribution

Things we still need to decide (DEFERRED):

  • Planned dishes flow: how does Mary "plan" a dish (say "Cam is bringing Greek salad") before Cam has posted? Is there an inline "Add planned dish" action on this screen? Or does planning happen elsewhere (event creation, attendee invitations)? Leaning toward: host can add planned placeholders during event creation AND from this screen; attendees can "claim" a planned slot when they post.
  • RSVP flow: where does RSVP live in the new model? The screen probably needs an "Are you coming?" state before the event happens, and "Are you here?" state during. Not drawn here because it's a flow not a screen.
  • Notifications: who gets notified when someone adds shared media? When a planned dish gets posted? When a comment is added to the evening? Separate work from this wireframe.
  • Eater rating privacy model carries over from F1e+ (D43/ζ). Not re-designed here — assumed to apply.
  • Migration path for existing meal posts: addressed in the earlier session — SQL migration renames post_type='meal' to post_type='meal_event', parent_meal_id threading is reinterpreted, post_participants with role sous_chef become cook_partners. Drawing-time reminder to verify when implementing.
