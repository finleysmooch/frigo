Input Relationships — What Feeds What
Session Log (repo)
feeds into →
All 4 Living Docs + Tracker + Feature Roadmap
The single capture point. Weekly sync reads entries and distributes changes everywhere.
Project Context
points to →
Architecture, Status, Deferred Work, Plans, Google Docs
Entry point. "Read me first, then go deeper." Every Claude session starts here.
Deferred Work
surfaces into →
Project Context ("Known Issues", "What's Next")
High-priority items surface into context doc. Full backlog stays in deferred work.
FRIGO_ARCHITECTURE (PK)
replaces →
Codebase Map (Google Doc — retired)
Single source of truth for code structure. More maintainable in project knowledge than Google Docs.
Product Architecture (Google Doc)
complements →
FRIGO_ARCHITECTURE (PK)
Domain definitions (product-level) + code map (code-level) = complete picture. Different audiences, different update cadences.
Feature Roadmap (Google Doc)
complements →
Deferred Work + Project Context
Product-level feature tracking with Notion links. Deferred Work has dev-level detail. Both track what's built and what's next.
DB Schema CSVs
inform →
Claude Shared Reference (Google Doc)
CSVs are raw data. Shared Reference is the annotated, readable version with purpose descriptions per table.
Git Commits
trigger updates to →
Architecture, Code files in PK, Tracker, Feature Roadmap
New files → architecture. Changed code → project knowledge. Features done → tracker + roadmap.
Weekly Sync — What Gets Updated
1. Read SESSION_LOG.md entries since last sync 5 min 2. Update 4 living docs in project knowledge (Context, Architecture, Status, Deferred) 15 min 3. Upload updated docs to Claude project knowledge, remove stale ones 5 min 4. Update FRIGO_TRACKER ↗ with committed code rows 5 min 5. Update Feature Roadmap ↗ (statuses, new features, file mappings) 5 min 6. Update Claude Shared Reference ↗ if DB schema changed 5 min if needed 7. Clear processed SESSION_LOG entries 1 min
Total: ~35-40 min with Claude
